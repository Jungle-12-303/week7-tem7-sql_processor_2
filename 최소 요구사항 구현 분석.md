# 최소 요구사항 구현 분석

기준: 현재 저장소 `/Users/juhoseok/Desktop/week7-tem7-sql_processor_2`

이 문서는 과제의 최소 요구사항이 현재 코드베이스 어디에 구현되어 있는지, 어떤 함수가 핵심 역할을 하는지 빠르게 찾기 위한 함수 단위 분석 문서입니다.

## 1. 전체 흐름

### INSERT 흐름

`sql_execute()`  
-> `sql_execute_insert()`  
-> `table_insert()`  
-> `bptree_insert()`

즉, SQL 처리기에서 INSERT를 파싱한 뒤 테이블에 레코드를 만들고, 그 순간 자동 증가한 `id`를 B+ 트리에 함께 등록합니다.

### SELECT WHERE id = ? 흐름

`sql_execute()`  
-> `sql_execute_select()`  
-> `table_find_by_id()`  
-> `bptree_search()`

즉, `WHERE id = ?` 조건은 선형 탐색이 아니라 B+ 트리 인덱스를 타도록 연결되어 있습니다.

### SELECT WHERE name/age = ? 흐름

`sql_execute()`  
-> `sql_execute_select()`  
-> `table_find_by_name()` 또는 `table_find_by_age()`

즉, `name`, `age` 조건은 인덱스 없이 배열을 순회하는 선형 탐색입니다.

## 2. 최소 요구사항 충족 여부

| 요구사항 | 상태 | 핵심 함수 |
| --- | --- | --- |
| 메모리 기반 B+ 트리 구현 | 구현됨 | `bptree_create()`, `bptree_insert()`, `bptree_search()` |
| leaf split / internal split | 구현됨 | `bptree_split_leaf_and_insert()`, `bptree_split_internal_and_insert()` |
| 레코드 INSERT 시 ID 자동 부여 | 구현됨 | `table_insert()` |
| INSERT 시 ID를 B+ 트리에 등록 | 구현됨 | `table_insert()`, `bptree_insert()` |
| `WHERE id = ?` 에서 인덱스 사용 | 구현됨 | `sql_execute_select()`, `table_find_by_id()`, `bptree_search()` |
| 다른 필드(`name`, `age`)는 선형 탐색 | 구현됨 | `sql_execute_select()`, `table_find_by_name()`, `table_find_by_age()` |
| SQL 처리기와 연동 | 구현됨 | `sql_execute_insert()`, `sql_execute_select()`, `sql_execute()` |
| 1,000,000건 이상 성능 테스트 | 구현됨 | `perf_test.c`의 `main()` |
| 인덱스 탐색 vs 선형 탐색 비교 | 구현됨 | `table_find_by_id()`, `table_scan_by_id()` |

## 3. 요구사항별 구현 위치

### 3.1 메모리 기반 B+ 트리 구현

파일: `bptree.c`

- `bptree_create_node()` (`bptree.c:6-15`)
  - 리프/내부 노드 공통 생성 함수입니다.
- `bptree_find_leaf()` (`bptree.c:29-50`)
  - 루트부터 시작해 특정 key가 들어갈 리프 노드를 찾습니다.
  - B+ 트리 탐색 경로의 핵심입니다.
- `bptree_insert_into_leaf()` (`bptree.c:70-82`)
  - 리프가 아직 가득 차지 않았을 때 정렬 상태를 유지하며 삽입합니다.
- `bptree_insert()` (`bptree.c:288-321`)
  - B+ 트리 삽입의 엔트리 포인트입니다.
  - 빈 트리 처리, 중복 key 방지, 일반 삽입, split 분기까지 담당합니다.
- `bptree_search()` (`bptree.c:325-340`)
  - `WHERE id = ?` 조회 시 최종적으로 호출되는 인덱스 검색 함수입니다.

### 3.2 leaf split / internal split

파일: `bptree.c`

- `bptree_split_leaf_and_insert()` (`bptree.c:147-199`)
  - 리프가 꽉 찼을 때 임시 배열에 key/value를 모아 정렬 후 좌우로 분할합니다.
  - `right_leaf->next = leaf->next; leaf->next = right_leaf;` 로 리프 연결 리스트도 유지합니다.
  - 부모에게는 오른쪽 리프의 첫 key를 올립니다.
- `bptree_split_internal_and_insert()` (`bptree.c:202-270`)
  - 내부 노드 overflow 시 가운데 separator key를 부모로 승격합니다.
  - 좌측/우측 자식들의 `parent` 포인터도 다시 맞춥니다.
- `bptree_insert_into_parent()` (`bptree.c:126-144`)
  - split 이후 승격된 key를 부모에 반영합니다.
  - 부모도 가득 찼다면 internal split를 재귀적으로 전파합니다.
- `bptree_create_new_root()` (`bptree.c:85-102`)
  - split가 루트까지 전파된 경우 새 루트를 만듭니다.

### 3.3 레코드 INSERT 시 ID 자동 증가 및 인덱스 등록

파일: `table.c`

- `table_create()` (`table.c:29-45`)
  - `table->next_id = 1;`
  - `table->pk_index = bptree_create();`
  - 테이블 생성 시 자동 증가 카운터와 PK 인덱스를 함께 초기화합니다.
- `table_insert()` (`table.c:65-96`)
  - `record->id = table->next_id++;` 로 ID를 자동 부여합니다.
  - `bptree_insert(table->pk_index, record->id, record)` 로 `(id -> Record *)`를 인덱스에 등록합니다.
  - 즉, 과제의 "레코드 추가 시 자동 ID 부여 + B+ 트리 등록" 요구사항을 직접 만족시키는 핵심 함수입니다.

### 3.4 `WHERE id = ?` 조건에서 B+ 트리 사용

파일: `sql.c`, `table.c`, `bptree.c`

- `sql_execute_select()` (`sql.c:214-289`)
  - `column`이 `"id"`인지 확인한 뒤 `table_find_by_id(table, int_value)` 를 호출합니다.
  - 여기서 SQL 레벨에서 인덱스 경로를 선택합니다.
- `table_find_by_id()` (`table.c:99-105`)
  - `bptree_search(table->pk_index, id)` 를 그대로 호출합니다.
  - 테이블 레이어에서 "PK 조회는 인덱스를 탄다"는 정책을 구현합니다.
- `bptree_search()` (`bptree.c:325-340`)
  - 실제 B+ 트리 탐색을 수행합니다.

### 3.5 `name`, `age` 조건에서 선형 탐색 사용

파일: `sql.c`, `table.c`

- `sql_execute_select()` (`sql.c:269-280`)
  - `name`이면 `table_find_by_name()`
  - `age`이면 `table_find_by_age()`
  - 즉, `id`가 아닌 컬럼은 테이블 스캔 기반 함수로 분기합니다.
- `table_find_by_name()` (`table.c:125-139`)
  - `rows` 배열을 처음부터 끝까지 순회하며 `strcmp()`로 찾습니다.
- `table_find_by_age()` (`table.c:142-156`)
  - `rows` 배열을 순회하며 같은 `age`를 찾습니다.

### 3.6 SQL 처리기와의 연동

파일: `sql.c`, `main.c`

- `sql_execute_insert()` (`sql.c:144-211`)
  - `INSERT INTO users VALUES ('Alice', 20);` 패턴을 파싱합니다.
  - 파싱 후 `table_insert()` 를 호출합니다.
- `sql_execute_select()` (`sql.c:214-289`)
  - `SELECT * FROM users;`
  - `SELECT * FROM users WHERE id = 1;`
  - `SELECT * FROM users WHERE name = 'Alice';`
  - `SELECT * FROM users WHERE age = 20;`
  - 위 네 가지 고정 패턴 SELECT를 처리합니다.
- `sql_execute()` (`sql.c:292-316`)
  - INSERT/SELECT/EXIT를 한 곳에서 분기하는 상위 엔트리 포인트입니다.
- `main()` (`main.c:9-64`)
  - 간단한 REPL입니다.
  - `sql_execute()` 결과에 따라 `table_print_all()` 또는 `table_print_record()` 를 호출해 사용자에게 결과를 보여줍니다.

### 3.7 대규모 데이터 삽입 및 성능 테스트

파일: `perf_test.c`

- `main()` (`perf_test.c:28-101`)
  - `PERF_INSERT_COUNT` 가 `1000000` 으로 설정되어 있습니다.
  - `table_insert()` 를 100만 번 호출해 대용량 데이터를 적재합니다.
  - 이후 `table_find_by_id()` 와 `table_scan_by_id()` 를 각각 1만 번 호출해 시간을 비교합니다.
- `now_ms()` (`perf_test.c:11-16`)
  - `gettimeofday()` 기반 시간 측정 함수입니다.
- `build_sample_ids()` (`perf_test.c:19-25`)
  - 성능 측정에 쓸 샘플 ID를 재현 가능하게 생성합니다.

주의:

- 현재 벤치마크는 "ID 인덱스 검색 vs ID 선형 검색" 비교입니다.
- 과제 문구를 아주 엄밀히 따르면 "ID 기준 SELECT vs 다른 필드 기준 SELECT" 비교도 가능하도록 `table_find_by_name()` 또는 `table_find_by_age()` 기반 벤치를 하나 더 두면 더 명확합니다.
- 다만 현재 코드도 "인덱스 탐색과 선형 탐색의 차이" 자체는 충분히 보여줍니다.
